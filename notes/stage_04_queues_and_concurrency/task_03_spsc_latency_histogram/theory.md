# Теория 3. SPSC Latency Histogram

## Throughput И Latency

Throughput отвечает на вопрос:

```text
сколько сообщений в секунду прошло через систему
```

Latency отвечает на другой вопрос:

```text
сколько времени одно конкретное сообщение провело в системе
```

В low-latency системах важны обе метрики. Можно иметь высокий throughput, но
плохой p99 latency, если иногда сообщения стоят в очереди слишком долго.

## Почему Total Time Недостаточно

Если producer отправил `1'000'000` сообщений, а весь run занял `100 ms`, можно
посчитать average cost per message.

Но это не говорит, что происходило с отдельными сообщениями:

- часть могла пройти быстро;
- часть могла ждать, пока consumer освободит очередь;
- часть могла попасть в scheduler pause;
- tail latency могла быть намного хуже median.

Поэтому для очередей часто смотрят histogram или хотя бы percentiles.

## Как Мерить Message Latency

Простой учебный способ:

```text
producer:
  message.sent_at = now()
  queue.push(message)

consumer:
  queue.pop(message)
  latency = now() - message.sent_at
```

Затем latency каждого сообщения добавляется в `LatencyStats`.

## Что Будет Шуметь

Такой benchmark не идеален:

- `steady_clock::now()` сам имеет стоимость;
- threads могут мигрировать между CPU cores;
- OS scheduler может остановить поток;
- consumer может отставать и измерять не только queue latency, но и backlog.

Но для учебной задачи это полезно: ты увидишь, что p99 почти всегда интереснее,
чем просто average.

## HFT-Мышление

В hot path важно не только "быстро в среднем".

Нужно спрашивать:

- что происходит с худшими сообщениями;
- откуда берутся spikes;
- растет ли p99 при заполнении очереди;
- можно ли уменьшить работу между receive и publish.
