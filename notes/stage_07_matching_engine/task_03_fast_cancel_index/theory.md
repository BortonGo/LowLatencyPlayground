# Теория. Fast Cancel Index

В текущей версии `cancel(order_id)` ищет ордер линейно:

```text
price levels -> deque orders -> compare id
```

Для маленького учебного стакана это нормально. Для matching engine это плохой hot path: cancel часто приходит так же часто, как add, а иногда даже чаще. Если каждый cancel сканирует всю книгу, tail latency быстро становится плохой.

## Идея Индекса

Чтобы ускорить cancel, нужно иметь быстрый путь от `order_id` к месту, где лежит ордер.

Самый простой первый шаг:

```text
order_id -> price
```

Такой индекс не дает сразу точный iterator на ордер, но уже позволяет не сканировать все price levels. Мы быстро находим нужный level по price, а внутри этого level ищем order id линейно.

```text
before:
all levels + all orders

after:
one level + orders inside this level
```

## Почему Не Сразу Iterator

Можно хотеть индекс:

```text
order_id -> iterator
```

Но с контейнерами нужно быть осторожным. Удаление элементов может инвалидировать iterator-ы или сделать поддержку индекса сложной. Для этой задачи важнее сделать понятную промежуточную оптимизацию и измерить эффект.

## HFT Мысль

Оптимизация начинается не с "перепишу все на массивы", а с вопроса:

```text
какая операция реально медленная и почему?
```

Наш benchmark уже показал: `cancel` сильно хуже `add` и `match`. Значит сначала улучшаем именно `cancel`.
