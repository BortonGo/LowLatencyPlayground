# Теория 2. Fixed-Size Memory Pool

## Зачем Нужен Memory Pool

В предыдущей задаче `new/delete` показал, что heap allocation в hot path может давать хвосты latency.
Один из типичных способов убрать эту проблему - заранее выделить память и во время обработки только брать готовый
slot из пула.

Fixed-size pool особенно удобен, когда все объекты одного типа или одного размера:

- message/event objects;
- order objects;
- preallocated buffers;
- nodes для очередей или списков.

Идея простая: в setup phase создается массив объектов или storage blocks, а runtime code только помечает slot как
занятый или свободный.

## Что Меняется По Сравнению С new/delete

`new/delete` каждый раз обращается к general-purpose allocator. Он должен уметь работать с разными размерами,
разными потоками, фрагментацией и metadata.

Fixed-size pool знает намного больше про задачу:

- размер объекта один и тот же;
- capacity известна заранее;
- память уже выделена;
- free slots можно хранить в простой структуре.

За счет этого acquire/release обычно дешевле и предсказуемее.

## Важный Компромисс

Pool не делает память бесконечной. Если все slots заняты, нужно явно решить, что делать:

- вернуть `nullptr`;
- бросить исключение;
- расширить pool;
- ждать освобождения slot.

В low-latency коде часто выбирают явное поведение без скрытого allocation. Для этой задачи лучше не расширять pool
автоматически: если capacity закончилась, benchmark должен это показать как ошибку логики.

## Что Важно Измерять

В этой задаче интересно сравнить:

1. `new/delete` для каждого объекта;
2. `pool_acquire_release` для заранее выделенного пула.

Смотри не только на mean. Для latency важнее понять, как отличаются median, p95, p99, p99.9 и max.

Если одиночные операции снова дают много `0ns`, это не обязательно ошибка. Это ограничение текущего таймера и очень
короткой операции. В таком случае можно сделать batch внутри measured iteration: выполнить несколько acquire/release
операций за один замер и интерпретировать результат аккуратно.
